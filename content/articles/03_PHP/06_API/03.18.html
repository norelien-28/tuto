3.2.18 – Versionning et bonnes pratiques REST

1. Pourquoi versionner une API ?

Le versionning permet d’apporter des changements sans casser les intégrations existantes des clients.
En effet, une API évolue avec le temps (nouveaux champs, suppression de fonctionnalités, modification de logique).

Objectifs du versionning :

  • Permettre l’évolution de l’API sans casser l’existant

  • Maintenir plusieurs versions en parallèle

  • Gérer des clients différents à des stades d’intégration différents

2. Méthodes de versionnement

a. Via l’URL (le plus courant)

bash
GET /api/v1/utilisateurs

Simple à implémenter, lisible et facilement versionnable.

b. Via l'en-tête HTTP Accept 

bash
Accept: application/vnd.monapi.v1+json

Plus propre sur le plan REST, mais plus complexe à gérer côté client et serveur.

c. Via un paramètre de requête (non recommandé)

pgsql
GET /api/utilisateurs?version=1

Moins élégant, moins RESTful.

3. Bonnes pratiques REST à appliquer

🔹 1. Utiliser les méthodes HTTP correctement

Méthode Action attendue
GET Lire une ressource
POST Créer une ressource
PUT Remplacer une ressource
PATCH Modifier partiellement
DELETE Supprimer une ressource

🔹 2. Utiliser les codes HTTP appropriés

Voir 3.2.16 – Gestion des erreurs et codes HTTP

🔹 3. Respecter les conventions d’URI

  • Utiliser des noms de ressources au pluriel :

    bash
    /api/v1/utilisateurs

  • Ne pas inclure de verbes dans l’URL :
    ❌ /getUsers
    ✅ /utilisateurs

  • Utiliser des sous-ressources :

    bash
    /utilisateurs/1/adresses

🔹 4. Réponses claires et cohérentes

  • Toujours retourner du JSON

  • Retourner un objet ou tableau structuré

  • Inclure des métadonnées si nécessaire (pagination, count…)

Exemple :

json
{
  "data": [
    { "id": 1, "nom": "Alice" },
    { "id": 2, "nom": "Bob" }
  ],
  "meta": {
    "count": 2
  }
}

🔹 5. Pagination des résultats

Nécessaire pour éviter les réponses trop lourdes :

bash
GET /utilisateurs?page=2&limit=10

Et dans la réponse :

json
{
  "data": [...],
  "meta": {
    "page": 2,
    "limit": 10,
    "total": 50
  }
}

🔹 6. Filtres et tri

  • Par nom :

    bash
    /utilisateurs?nom=Alice

  • Tri :

    bash
    /utilisateurs?sort=nom&order=asc

🔹 7. Séparation claire des rôles

  • Authentification : /login, /logout

  • Ressources sécurisées : /utilisateurs, /commandes

  • Permissions selon rôles (admin, user...)

🔹 8. Documenter l’API

  • Swagger/OpenAPI

  • README clair avec :

    • Endpoints

    • Paramètres attendus

    • Exemples de réponse

    • Codes d’erreur

4. Exemple de versionning simple en PHP (via URL)

Structure :

bash
/api/v1/index.php
/api/v2/index.php

Dans chaque version, tu peux avoir ta logique et ta structure indépendamment.

5. Quand créer une nouvelle version ?

  • ⚠️ Tu changes une structure de réponse

  • ⚠️ Tu supprimes ou renommes un champ

  • ✅ Tu ajoutes des champs facultatifs ➜ pas besoin de nouvelle version
